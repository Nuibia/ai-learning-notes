# 我终于分清了 Agent Loop 的模型上限和工具执行

今天我继续学习最小真实 Agent Loop。以前我知道它大致是“用户请求 → Runtime 调模型 → 模型要用工具就执行 → 再调模型”，但我没有真正分清循环到底由谁控制，以及 `usedTurns` 和 `maxTurns` 放在哪里才合理。

这次我把链路还原成了一个 Runtime 主导的状态机：

1. Runtime 根据用户请求组装 `messages`。
2. 每次准备调用 LLM 前，先检查 `usedTurns >= maxTurns`。
3. 未达到上限时调用 LLM，并把 `usedTurns` 加一。
4. LLM 返回合法 `final`，Runtime 直接返回 `final.answer` 并结束。
5. LLM 返回合法 `tool_call`，Runtime 校验参数、执行工具，并把实际结果作为带相同 `tool_call_id` 的 `role=tool` 消息回填到 `messages`。
6. 回到下一次 LLM 调用前的上限检查。

我今天真正纠正的是：`usedTurns` 在这份课程合同中只表示已经发起的 LLM 调用次数，不是工具调用次数。

例如 `maxTurns=2` 时，第二次 LLM 调用如果仍返回合法 `tool_call`，按照这份合同，Runtime 仍会执行工具并回填结果；之后准备第三次 LLM 调用时，才发现 `usedTurns` 已达到上限，因此以 `loop_limit` 结束。它限制的是模型调用，不是工具动作。

这也让我看到一个真实边界：若最后一次 LLM 请求的是“发送邮件”这类写入工具，单靠 `maxTurns` 并不能阻止它执行。生产系统还需要独立的工具预算、权限确认，或者“剩余模型轮次不足时拒绝写入工具”的策略。不能把“模型可以请求工具”误解为“Runtime 应自动执行所有工具”。

我今天亲自写出了三条 Trace：第 1 轮直接 `final`、工具成功回填后第 2 轮 `final`、以及两次 `tool_call` 后因模型调用上限停止。现在我能把模型输出、工具回填、调用计数和终止原因放到同一条链路里理解。

下一步，我会不看笔记重新画出这三条状态图，并为一个写入工具补一条独立的权限或预算规则。
